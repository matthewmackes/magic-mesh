## MESH-BRINGUP — findings from the live 4-node bring-up (2026-06-13)

- [✓] **MESH-1 · DESIGN/SEC · enrollment has no network bootstrap — remote/NAT'd peers can't self-join.** `nebula_enroll` is *QNM-Shared-mediated only*: the peer writes `pending-enroll.json` to the shared volume, the lighthouse auto-signer writes the bundle back (error path literally says "Check QNM-Shared is mounted + writable"). ## XCP-PROVISION-AUDIT — findings from spinning up 4 MDE-VMs + installing the platform (goal 2026-06-16)
Live audit of the clone→boot→install→enroll flow on the XCP host; each becomes an XCP-epic acceptance hardening item.
- [ ] **XPA-1: default VM memory (4 GB) overcommits a typical host.** A 15.9 GB host couldn't start 4×4 GB VMs + the base. Right-size headless Server VMs to **2 GB** by default in the provisioner (XCP-3), and pre-check host free memory before fan-out (fail early with a clear message, not "no suitable hosts").
- [ ] **XPA-2: clone-from-snapshot needs the snapshot UUID (snapshots aren't in `xe vm-list`).** The golden must be a real **template** (`xe template-list`) per XCP-2, and the provisioner resolves it by template name → uuid; don't clone a name via vm-list.
- [ ] **XPA-3 (CRITICAL): cloud-init does NOT re-run on a cloned *booted* image → identity not reset.** All 4 clones kept the base's hostname (`f44-test`), machine-id, and SSH host keys despite a fresh NoCloud seed (new instance-id). The golden MUST be generalized before cloning (`cloud-init clean --logs --seed`, truncate `/etc/machine-id`, remove `/etc/ssh/ssh_host_*`) so each clone regenerates identity on first boot — OR the provisioner does an explicit post-clone identity reset (hostname + `systemd-machine-id-setup` + `ssh-keygen -A`). Build this into XCP-2 (golden) + XCP-3 (spawn).
- [ ] **XPA-4: clone copies the VIF MAC → collisions.** The provisioner must reset each clone's VIF (new MAC) — fold into XCP-3.
- [ ] **XPA-5 (suspected, verifying): enroll-token endpoint is the OVERLAY IP.** `mackesd enroll-token` mints `mesh:…@10.42.0.2:4242#…` (the Nebula **overlay** IP), which a not-yet-enrolled LAN/internet node can't reach. The join token for an external node must carry a **publicly/LAN-reachable** enroll endpoint (the relay's public IP / the v3 `/enroll` HTTPS on :443). Confirm + fix the token-minting endpoint selection.
- [ ] **XPA-6: the monolithic RPM bundles all GUI binaries (~100 MB) — a headless Server install carries surfaces it never runs.** The full `cargo build --workspace --release` (libcosmic GUIs: workbench/music/voice-hud/applet) gates every install + a 100 MB+ transfer per node. Consider a `magic-mesh-server` subpackage (daemon + nebula + lizardfs-client, no GUI bins) for headless roles, or split GUI assets behind the role so a Server pulls only what it runs.
