## Chat / Messaging (NOTIFY-CHAT: mde-chat model + mackesd `chat` worker + mde-shell-egui `Surface::Chat`)

The Chat/Messaging subsystem is the "ICQ-style Mesh Chat" unified messaging + notification interface (design: docs/design/mesh-chat-icq.md, 25 locks). It has three tiers. (1) A pure, headless model crate `crates/services/mde-chat` (lib/message/conversation/roster/notify/alert): six MessageKinds (Text/Clipboard/Alert/File/CallAction/RemoteAction), Ed25519-signed messages, a bounded (cap 500) append-only ring `Conversation` ordered by (ts,sig,id) with idempotent-by-id insert, `Room`/`RoomDescriptor` (ad-hoc + auto system rooms: All Fleet + per-severity), `Roster`/`Contact`/`Presence`/`NodeRole`, `NotifyPrefs` (per-contact/room/source mute + severity threshold + DND gate), and the pure `fold_alert` that turns any Bus alert lane into an Alert message from its origin host. (2) The universal (rank-0, every node incl. headless) mackesd `chat` worker `crates/mesh/mackesd/src/workers/chat.rs` (2772 lines): drains action/chat/* verbs, signs+relays on event/chat/message, writes each host's own outbound ring to `<workgroup>/<host>/chat/out/<key>.json` for Syncthing import-union backfill + offline queue, folds ALERT_LANE_PREFIXES into `alert:<host>` + per-severity rooms, derives presence from mesh-status + gossiped overrides, and republishes state/chat/{roster,rooms,notify,conversation/<key>} read-models + KIRON toasts. (3) The pure-renderer surface `crates/desktop/mde-shell-egui/src/chat/{mod,render}.rs` (~3500 lines): ICQ roster rail (Online/Offline groups, self presence picker + status editor), rooms group + create/join/dissolve, per-contact conversation panes with composer/Send-To/Call/Remote/mute/delivery-state, aggregate Notifications lane (DND/threshold/source-mute), typed alert-action buttons (safe/armed-destructive/ack/snooze), feed filters, unread watermarks feeding the dock badge + Start-menu tile, and its own Contacts/View/Chat/Presence/Help menu bar. There is NO `mackesd chat` CLI subcommand and NO message edit/delete (the ring is append-only — edit/delete windows are net-new in Communications). Cross-surface producers (Files, Browser, Terminal) publish action/chat/send into it. It maps into Communications modes Messages/Alerts/Clipboard/Files/Transfers/Calls/Activity with the space-rail + presence projection replacing the roster.

## Voice / Calls / SIP

The Voice/Calls/SIP subsystem is two loosely-coupled halves plus a legacy third. (1) The SOFTPHONE: a standalone/embedded egui surface `mde-voice-egui` (E12-11) rendering a Dialer + a Fleet-admin tab, driven by three in-process worker threads over the pure-Rust SIP core in `mde-voice-hud` — `sip.rs` (REGISTER w/ RFC2617/7616 digest over UDP + TLS, INVITE/ACK/BYE, inbound INVITE/BYE agent, re-REGISTER backoff), `media.rs` (RTP/G.711 µ-law/A-law + RFC4733 DTMF + mic-mute over cpal), `secure.rs` (TLS/UDP transport policy + per-node reg-state JSON), `roster.rs` (mesh-peer roster loader — now orphaned), `resolve.rs` (1NNN/9+11 dial heuristic — now orphaned). The surface mounts in `mde-shell-egui` as `Surface::Voice` (dock Comms group, Num5/F5 car-HUD route, launcher pin, VoIP HUD tile). (2) The FLEET PROVISIONING plane: mackesd's leader-gated `voice_provision` worker (VOIP-GW-3) drives the typed `vitelity` client (VOIP-GW-2) to give each enrolled node a Vitelity sub-account, seals creds to the secret store, routes DIDs, sets failover, holds the shared-outbound trunk, and drives the legacy→split cutover — all surfaced through the Fleet tab's `action/voice/*` verbs and `state/voice/*` topics; plus the mesh-wide SIP gateway responder (`action/voip/*`, VOIP-GW-1) and Vitelity-link RTT telemetry (`voip/link-rtt/*`, VOIP-4). (3) The LEGACY mesh-PBX config stack: `mde-voice-config` generates Kamailio/RTPengine configs, materialized to `voice-desired.json` by `voice/materialize`, reloaded by the `voice_config` worker via `mackesd voice render-config`, with `voice/mod` best-path routing — this server-daemon path appears superseded by the pure-Rust softphone + per-node Vitelity model and needs an explicit keep/retire ruling. Persisted state a migrator must import: `~/.config/mde/voice/account.toml`, `<workgroup>/voip/gateway.toml`, `<workgroup>/voip/peers.toml`, the secret-store voice/* keys, and the enrolled `nodes` SQLite table.

## Editor (mde-editor-egui) — the "Construct" native code-editor surface

The Editor is a Zed/Word-97-hybrid native code editor living entirely in crates/desktop/mde-editor-egui (27.3k LOC across 33 files) and mounted in the one DRM-native shell (crates/desktop/mde-shell-egui/src/main.rs:1401 as Surface::Editor via real_editor()/editor_panel()); there are NO mackesd CLI verbs, workers, or SQLite tables for it. ## File manager (Surface::Files) — mde-files (services lib) + mde-files-egui (desktop surface) + mackesd mesh_mount/transfers/mesh-ssh-key workers

The File manager is a two-crate desktop file-manager plus a mesh backend. mde-files (crates/services/mde-files) is a render-agnostic library: a Backend trait (RealBackend = local FS + mesh Bus, LocalFsBackend, legacy DemoBackend removed), a full-POSIX FileOps trait (copy/move/rename/delete/mkdir/new-file/symlink/hardlink/chmod/chown/set-times/duplicate), an async op queue (opqueue: Copy/Move/Delete/Compress/Extract with a conflict engine + progress/pause/cancel), archives (zip/tar/gz/xz/zst), recursive name+content search, a direct A→B transfer router, and Bus clients (fleet-files, nebula, editor-open, send-to). mde-files-egui (crates/desktop/mde-files-egui) is the FILEMGR-8 egui surface (FileBrowser model + view + menubar + dialogs + preview + transfers/mesh_mount/chat clients), mounted by mde-shell-egui as Surface::Files and also as a standalone binary. The mesh tier is three mackesd workers: mesh_mount (FILEMGR-5, sshfs lifecycle over action/mesh-mount + state/mesh-mount), transfers (TRANSFERS-1, a file-store ledger/inbox queue with sftp/rsync/http/browser-download/node/music lanes) and sshd_overlay_bind + the mesh-ssh-key CLI (FILEMGR-6). Reachable surface capabilities: List/Grid/Details views, click-header sort, dirs-first, show-hidden, breadcrumbs + editable path box + back/forward/up history, tabs, dual-pane, Places+Mesh sidebar, click/ctrl/shift/ctrl-A/rubber-band selection, drag-drop move/copy, mesh peer mount/escalate/unmount, thumbnails+preview pane+quick-look (built-in image/text/media viewers), recursive search with filters, permanent-delete-with-confirm (typed-arming on remote/escalated mounts), Properties (chmod/chown), Send-To/Send-in-Chat/Send-to-Editor, shared cut/copy/paste clipboard, and a Transfers tab (New Transfer + pause/resume/cancel/clear). NOTE: rename/mkdir/new-file/symlink/hardlink/duplicate/compress/extract are implemented in the backend (FileOps + OpKind) but have NO reachable UI command (menubar.rs explicitly bans New Folder/Rename); and the legacy Cosmic-era views (Inbox/Outbox/Downloads/MeshHome/Recycle-Bin/Cloud-Files/Network) in mde-files/model.rs are dead in the egui surface.

## Transfers (TRANSFERS-1..11 — the daemon-owned universal file-move queue/ledger + its Files-tab renderer + the browser download manager)

Transfers is "the one place every byte that moves is born, tracked, and completed" (docs/design/transfers-surface.md, 20-Q lock). Execution lives entirely daemon-side in the mackesd rank-1 `transfers` worker (crates/mesh/mackesd/src/workers/transfers/): a typed TransferJob envelope + five-state machine (job.rs), a persistent node-local JSON ledger (ledger.rs), a synchronous queue engine with a configurable parallel cap and optional SHA-256/size integrity verify (queue.rs), a typed verb set carried over a node-local file inbox (verb.rs), durable recurring rsync sync pairs + scheduler (sync_pair.rs + mod.rs), an auto destination registry projected from the peer roster (destination.rs), and six execution lanes — sftp/http-wget/rsync/browser-download/node/music — behind an injectable LaneRunner seam (lane/mod.rs, browser_media.rs, helpers.rs). §9 doctrine keeps the daemon the single ledger writer; the GUI only submits verbs + reads the ledger. Two renderers front it: the Files browser's Transfers tab (mde-files-egui: transfers.rs client-mirror + model/mod.rs + view.rs + menubar.rs) with three entry points (New Transfer dialog / drag-drop / Send-to→), and the browser chrome's Downloads drawer (mde-shell-egui/src/web) which folds every browser download+scrape output into the same ledger as `browser_download` jobs (Q8/Q17). A `mackesd transfer` CLI (cli/transfer.rs) gives §9 parity. Terminal jobs emit on event/notify/transfers, folded by the chat worker into the node's alert:<self> feed. State is node-local files under $MDE_HOME/transfers (ledger/, inbox/, sync-pairs/). A separate FILEMGR-7 mesh_transfer IPC (ipc/mesh_transfer.rs) is the File-Manager's direct peer↔peer rsync engine — adjacent, not part of this ledger. Two notable gaps: the SaveSyncPair/RemoveSyncPair verbs have NO reachable producer (recurring pairs are code-complete but unreachable), and there are no dedicated keyboard hotkeys.

## Alerts / Notifications

Today the platform has NO standalone Notifications surface — it was retired (BUS-4.2 / NOTIFY-CHAT-6) and unified into "the ONE notification interface": the Chat surface. The live pipeline is: (a) emitter workers publish alert-shaped JSON onto Bus alert lanes (event/security, fleet/sec, event/firewall, compute/event, event/kvm\|vm\|dc\|podman, fdo/, event/notify/, fleet/health/); (b) the mackesd `chat` worker's drain_alerts folds EVERY such lane via mde_chat::fold_alert into per-host `alert:<host>` conversations + per-severity system rooms (Critical Alerts/Warnings/Info), gated by NotifyPrefs (per-contact/room/source mute + severity threshold + DND), and raises a Warning+ chyron on event/toast/show; (c) the shell renders these in crates/desktop/mde-shell-egui/src/chat (roster, aggregate Notifications lane, alert cards with ack/snooze/safe/armed-destructive actions, mute + DND + min-severity controls, unread badge). A parallel headless path: the `notify` worker (CHAT-FIX-2) produces local system events (peer join/leave, updates, failed service, disk/SMART, journal WARN+, node-grade D/F, cloud) onto event/notify/<source> + publishes state/notify/segment/* rollups that drive the dock's 4 health pips (status.rs, NOTIF-3) and the own-seat critical screen-edge cue (NOTIF-6). ## Clipboard sync

The magic-mesh "Clipboard sync" subsystem is three independent-but-overlapping lanes plus the surfaces that touch them. (1) The MESH-GLOBAL TEXT CLIPBOARD (CLIP-SYNC-1/2): the `clipboard_sync` worker (Workstation rank-1) spawns `wl-paste --watch` — discovering the active seat0 graphical session via logind when running as the root daemon — captures every text selection, folds it through the pure `apply_clip` policy (O2 debounce echo / O3 dedup move-to-top / O6 source+timestamp stamp / O7 pins exempt from a 50-cap), writes ONE Syncthing-replicated `<workgroup_root>/clipboard/history.json`, and broadcasts each clip on `event/clipboard/clip`; the `clipboard_bus_responder` thread serves `action/clipboard/{list,pin,unpin,delete,clear}` against the same file (content-addressed SHA-256 `clip_id`, mesh-wide delete/clear, unlimited pins). (2) The VDI CLIENT↔GUEST BRIDGE (E12-9): the `clipboard_bridge` worker (rank-0, node-local) drains `action/vdi/clipboard`, applies a per-session `ClipboardPolicy` (allow/deny + one-way + 1 MiB cap → Forward/Drop/Truncate), MIME-tags payloads (`ClipFormat::mime`), and relays through the injectable `ClipboardAccess` seam — whose production `OsClipboardAccess` is integration-gated (SPICE/RDP vdagent / wl-clipboard never wired).
