# magic-mesh — Platform Review

_Senior-architect + product/UX review of the whole platform (~558K LOC, 41 crates). Findings produced by an 11-lens fan-out review, each finding adversarially verified against real code by an independent second agent before inclusion. Dated 2026-07-10._

**85 findings verified-confirmed** + 27 raised-but-unverified (verifier rate-limited; shown separately, marked ⚠︎UNVERIFIED). Priorities calibrated by the verifiers: P0×0, P1×21, P2×45, P3×19.

## Executive summary

magic-mesh is an ambitious, largely-working DRM-native egui VDI desktop OS whose engineering discipline is real (typed error handling on untrusted paths, headless folds, compile-time completeness guards) but which carries three classes of enterprise-grade risk that cluster tightly. First, SECURITY & FLEET-STABILITY defects with live blast radius: the DEFAULT browser engine (CEF) runs untrusted web content with Chromium's sandbox disabled, no OS confinement, and an unauthenticated DevTools port on a node holding the Nebula CA and SSH keys (security-1/-4); WebAuthn ceremonies complete with a forged User-Present bit and no consent (security-2); and seven leader-gated workers still elect on a per-node Syncthing-replicated lockfile after election moved to etcd, so every node self-elects and runs exactly-once work (DR backups, billable SIP provisioning, snapshot retention) N times (mackesd-01/-04). Second, the FLAGSHIP VDI PATH is incomplete and incorrect where it matters most: a peer's own KVM VMs can never attach a live transport (loopback-bound, no port broker — vdi-vm-1), guest pointer input is never transformed from panel geometry to desktop pixels so clicks miss in essentially every session (vdi-vm-2), any transport drop freezes the desktop with no reconnect and no error banner (shell-ux-1/vdi-vm-4), and the same coordinate bug afflicts the browser (browser-1). Third, PERFORMANCE ON THE SHIPPED DRM SEAT is structurally wasteful: the present loop renders the full shell at refresh rate forever (making all repaint-throttling dead code — perf-1), the seat snapshot runs pw-dump + a blocking DDC/CI I2C probe on the render thread every 5s (guaranteed periodic freezes — perf-2), and dozens of pollers re-open SQLite and re-read entire topic histories per tick just to take the newest row (perf-3/-4). Underneath sits a maintainability tax — two ~20K/12K-line god-files (web.rs, mackesd.rs) and pervasive drift-prone duplication — plus an accessibility posture that looks invested but is unreachable in production (all AccessKit enablement is test-only; the DRM seat has no assistive-technology path at all — a11y-01/-02), and front-door docs that still describe the retired COSMIC/iced product. Biggest opportunities: (1) close the browser-security and split-brain gaps before any wider deployment; (2) finish and correct the VDI core so the product's central promise actually works; (3) make the seat event-driven and I/O-light; (4) collapse the duplicated leadership/wire/VDI/publish logic into single sources of truth, which simultaneously fixes whole bug classes and unblocks the god-file splits.

## Cross-cutting themes

- God-files & monoliths: web.rs (20,107 lines), mackesd bin (12,161 lines / run_serve ~3,970), and a 1.49MB WORKLIST defeat review, incremental builds, and parallel-agent work (arch-1, arch-3, mackesd-07, shell-ux-7, browser-2, docs-consistency-10)
- Duplicated logic that drifts silently: leadership is_leader() copy-pasted 7-15x (root cause of the etcd split-brain), ~40 hand-mirrored JSON wire structs with no shared contract, triplicated VDI input/framebuffer code, and 3-4 divergent Bus-publish idioms (mackesd-01/-04, arch-2, arch-6, arch-8, arch-9, test-obs-8)
- Accessibility invested but unreachable: every enable_accesskit() sits in #[cfg(test)]; the shipped DRM seat has no AccessKit consumer and no AT-SPI/screen-reader/TTS path; keyboard focus is invisible on painted cells; largest surfaces carry zero semantics (a11y-01..09, shell-ux-6)
- Flagship VDI path incomplete or incorrect: mesh VM brokering never attaches pixels, guest input mis-mapped, fixed 1024x768 upscaled, no client reconnect, migration undefines source before target confirms (vdi-vm-1..8, shell-ux-1, browser-1)
- Browser security posture: default engine (CEF) unsandboxed with no SELinux domain, unauthenticated CDP:9222, iframe-bypassable WebRTC block, passkey shim forges User-Present and breaks standard Credential Management (security-1/-2/-4, browser-3/-4/-6)
- Hot-path inefficiency on the DRM seat: full-shell render at refresh rate, blocking DDC/pw-dump probes on the render thread, per-tick SQLite opens, full-topic-history reads for latest-wins, full-framebuffer uploads discarding dirty rects (perf-1..11)
- 'Compiles != ships != reachable' release-integrity gaps: surface mount + RPM asset lists wired in N unlinked places (already shipped dead TERM/MEDIA epics and a GUI-less 11.4.0 RPM); Explorer built but unreachable (arch-12, shell-ux-8)
- Docs describe a retired epoch: README/architecture/help/build docs still teach COSMIC/iced/Carbon/mde-workbench; brand ships as both Quazar and Quazar; CHANGELOG dead since 11.3.1 (docs-consistency-1..10, shell-ux-9)

#### `browser-1` Browser renders at a hardcoded 1280x800 that never resizes to the viewport, and pointer input is mis-scaled against the fixed frame
**[P1 · M · risk med]** — `crates/desktop/mde-shell-egui/src/web.rs:120-122,3980-3981,10342-10403; crates/desktop/mde-web-preview-client/src/input.rs:27-41`

- **Problem:** Tabs spawn at a fixed INIT_W=1280 x INIT_H=800 (web.rs:120-122, SpawnSpec at 3980) and WebSession::resize / ControlMsg::Resize is never called anywhere in the shell (grep for .resize(/session.resize/ControlMsg::Resize across the shell crate returns zero call sites), so the page CSS viewport is permanently 1280x800 regardless of the real panel/display size. The CEF frame channel capacity is fixed at creation (offscreen.rs:57-65, shm.rs FrameChannel::create) and resize() only updates atomics (cef_browser.rs:1109-1114); an on_paint larger than the initial capacity is silently dropped (publish returns TooLarge, swallowed by 'let _ =' at cef_browser.rs:1510). Pointer input is mapped as (pos - image_rect.min) * pixels_per_point (browser_input_event web.rs:11350-11376 -> input.rs map_event pos.x*ppp), which equals the 1280x800 frame device coordinates only when the image is displayed at exactly native resolution. The correct frame-space mapping (rel * frame_size / image_rect.size) already exists but is used ONLY by region-capture (web.rs:10417-10423), not the live input path.
- **Why it matters:** This makes the browser materially broken on real seats. On any display where the fit-to-aspect image_rect is not exactly 1280x800 device px (HiDPI seats, maximized windows, 4K displays, non-1.6-aspect panels), clicks land at the wrong page coordinates (systematically stretched/compressed), edge links/fields become unhittable, and pages get a fixed 1280x800 responsive breakpoint that is upscaled (blurry) on big displays and downscaled (tiny text) on small ones. For a VDI thin-client whose flagship surface is the browser, this is a top-tier usability and correctness defect.
- **Recommendation:** Drive session.resize(device_w, device_h) from paint_body using the allocated image rect in device pixels whenever it changes (debounced), map pointer coordinates into frame space with the same rel*frame_size/image_rect.size transform region-capture already uses (fold both paths through one helper), and make the CEF/Servo frame channel reallocate (or pre-size to the seat's max) on grow so enlarged paints are not dropped.
- **Expected benefit:** Correct hit-testing, crisp native-resolution rendering, and correct responsive layout across all seats and window sizes; retires a large class of 'links do not click' / 'page is blurry' complaints.
- **Verifier note:** Verified all cited evidence: INIT_W/INIT_H=1280x800 at web.rs:120-122 feed SpawnSpec (~3977-3981) and WebSession::resize (session.rs:594) has zero call sites in mde-shell-egui; paint_body fit-scales the texture (web.rs:10355) while browser_input_event + input.rs map_event send only (pos - rect.min) * ppp, which matches frame space solely at native 1:1 display scale — the correct rel*frame_size/image_rect transform exists only in region-capture (web.rs:10417-10423); CEF frame channel capacity is fixed at attach (cef_browser.rs:1084, shm.rs:135-144), resize only stores atomics (cef_browser.rs:1109-1113), and publish TooLarge is swallowed by let _ = (~1511) — so clicks are systematically mis-registered on essentially every real seat (chrome height alone breaks the one exact geometry), a genuine P1 critical defect on a flagship surface, though not P0 since no crash/data-loss/security impact.

#### `vdi-vm-2` Guest pointer input is not transformed from panel geometry to desktop pixels — cursor is misaligned in every session
**[P1 · M · risk low]** — `crates/desktop/mde-shell-egui/src/vdi.rs:1090-1097,1170-1211; crates/desktop/mde-vdi-spice/src/input.rs:332-347`

- **Problem:** The desktop texture is painted scaled to `ui.available_size()` at a rect with a non-zero origin (vdi.rs:1090-1092), and the negotiated desktop is a fixed 1024x768 (RDP `.with_resolution(1024,768)`, SPICE `.with_size(1024,768)`). `forward_input` forwards the raw egui events unchanged (vdi.rs:1184-1210), and `map_event` casts the egui pointer position directly to desktop pixels via `clamp_u16(pos.x)`/`clamp_u16(pos.y)` (spice input.rs:334-347; RDP/VNC input mappers are the same shape). There is no subtraction of the painted rect's top-left origin and no scale factor from the painted rect size to the guest resolution. The guest cursor therefore lands at the wrong coordinate whenever the panel size != 1024x768 or the desktop image is not at window origin (0,0) — i.e. essentially always, since a menubar/dock sits above it and the panel is sized to the seat.
- **Why it matters:** Clicks and drags miss their targets in the remote desktop, which makes the VDI surface unusable for real work regardless of which transport is attached (all three consume the same forward_input path). This is a correctness defect, not a cosmetic one.
- **Recommendation:** In forward_input, translate pointer positions from egui screen space into desktop pixels before enqueueing: subtract the painted rect origin, then multiply by `desktop_size / rect.size()` per axis, and clamp to the guest desktop bounds (not u16::MAX). Pass the rect + current desktop_size into the input path. Ideally also negotiate the desktop at the panel's real size and re-negotiate on resize (see fixed-resolution issue) so the scale is 1:1.
- **Expected benefit:** Pointer clicks land where the operator aims; the desktop becomes genuinely interactive.
- **Verifier note:** Verified end-to-end: vdi.rs:1090-1092 paints the desktop scaled into an offset rect (menubar above at main.rs:588-596), forward_input (vdi.rs:1184-1210) forwards raw screen-space egui events through mpsc to the workers unchanged, and all three transport mappers (spice input.rs:334-347, vnc input.rs:278-291, rdp input.rs:315-328) cast pos.x/pos.y directly to u16 guest pixels with no origin subtraction or scale — against a fixed 1024x768 negotiation (vdi.rs:368,501) — so guest pointer input is misaligned in essentially every session; the sibling web surface already passes a scale factor (web.rs:10393), confirming the omission.

#### `build-deploy-1` Live root password committed to git and pushed to GitHub ⚠︎UNVERIFIED
**[P0 · S · risk low]** — `docs/WORKLIST.md:4149 (tracked at HEAD; remote = https://github.com/matthewmackes/magic-mesh.git)`

- **Problem:** The plaintext credential 'mm/<REDACTED>' — the real root/login password for the physical seat machines (172.20.146.2, and per operator notes also .138 and Eagle) — is embedded in the SEAT-1 acceptance text of docs/WORKLIST.md, committed at HEAD and pushed to the GitHub origin. The same repo's gh-pages branch hosts the fleet's public dnf channel (packaging/repo/magic-mesh.repo baseurl=https://matthewmackes.github.io/magic-mesh/...), which requires GitHub Pages and strongly implies public visibility of the repository containing the password.

- **Expected benefit:** Eliminates a standing P0 credential exposure and prevents recurrence of the credentials-in-docs pattern.

#### `mackesd-01` Split-brain: leader-gated action workers elect on the fs lockfile while SUBSTRATE-V2 moved election to etcd
**[P1 · M · risk med]** — `crates/mesh/mackesd/src/workers/{dr_scheduler.rs:172, dc_auditor.rs:321, voice_provision.rs:1199, session_broker.rs:1045, copilot.rs:1216, dc_snap_scheduler.rs:634, upgrade_intent_watcher.rs:389} vs leader_election.rs:161-215`

- **Problem:** Seven leader-gated action workers gate their real work on self.is_leader() = crate::leader::try_acquire(&self.leader_lock, ...), the advisory-lockfile election on <workgroup_root>/.mackesd-leader.lock. But on a SUBSTRATE-V2 node the leader_election worker takes its etcd branch (leader_election.rs run(): if self.uses_etcd() it only calls campaign() and NEVER calls tick_once()), so the fs lockfile is never renewed. Grep confirms all seven workers have ZERO etcd awareness (0 hits for substrate::leader/etcd_client). Two consequences on the live etcd+Syncthing fleet: (a) the fs lease is permanently stale/absent, so every node's try_acquire returns Acquired; and (b) the workgroup root is a per-node Syncthing replica (mackesd.rs run_serve: 'the workgroup root is the plain Syncthing directory at /mnt/mesh-storage'), where fs2 POSIX advisory locks do not coordinate across hosts. Every node therefore self-elects as leader.
- **Why it matters:** These are the workers whose whole point is 'run exactly once mesh-wide.' Split-brain means: dr_scheduler runs the DR backup on every node; dc_snap_scheduler takes duplicate xe vdi-snapshots and each node's retention pass destroys snapshots simultaneously (race/double-destroy); voice_provision creates a Vitelity SIP sub-account per node (duplicate EXTERNAL, billable accounts); copilot fires the codex/LLM ask N times; dc_auditor/dc_jobs write duplicate audit rows. The mackesd.rs:7702-7708 spawn comment still asserts the stale premise ('the shared QNM-Shared mount makes them contend for the same lock').
- **Recommendation:** Introduce one leadership resolver (e.g. leader_gate(workgroup_root, node_id) that branches on substrate::etcd::default_endpoints() exactly like health_reconciler/etcd_watch/directory already do) and route all seven is_leader() sites through it, so etcd nodes gate on substrate::leader::current_leader/campaign and only pre-cutover nodes use the fs lock. Until then the etcd-fleet leader-gated workers must not run on the fs path.
- **Expected benefit:** Restores single-leader semantics for DR, snapshots, external SIP provisioning, and copilot on the live etcd fleet; eliminates duplicate billable accounts and snapshot-retention races.
- **Verifier note:** Verified all cited evidence: leader_election.rs:162 etcd branch never renews the fs lock, all seven workers gate on crate::leader::try_acquire with zero etcd awareness (grep-confirmed), mackesd.rs:7706-7707 stale shared-mount comment is verbatim, and the workgroup root is a per-node Syncthing replica (mackesd.rs:6387) where fs2 advisory locks give no cross-host exclusion — so multi-leader execution of exactly-once work (Vitelity provisioning, snapshot retention, DR backups) is real; minor correction: workers' own try_acquire calls create local leases (not "permanently absent"), and partial reconcile idempotence keeps this P1 not P0.

#### `mackesd-02` Unbounded blocking subprocesses run inline on the tokio runtime threads
**[P1 · M · risk low]** — `crates/mesh/mackesd/src/workers/dr_scheduler.rs:108-138 (run_backup) + run()@~204; pattern-wide (184 bare std::process::Command call-sites)`

- **Problem:** dr_scheduler.run() calls run_backup(&script) directly inside the async run() loop (not spawn_blocking), and run_backup shells `bash -lc <dr-backup.sh>` with a bare std::process::Command::output() and NO timeout. A DR backup (rsync/tar of the fleet) can take minutes or hang; it blocks a tokio worker thread for its entire duration. The runtime is floored at only 4 worker threads (WATCHDOG-2), and the watchdog liveness beat rides that same runtime, so a long/hung backup pins 25% of the executor and can starve the beat. This is not isolated: 184 bare blocking Command call-sites exist across the workers vs only 29 files that use spawn_blocking, and proc.rs (EFF-20) itself documents that sync tick helpers 'run directly on the tokio runtime thread ... a hang pins a whole runtime worker indefinitely.'
- **Why it matters:** On a 1-vCPU lighthouse (floored to 4 threads) a few concurrent blocking shell-outs can wedge the whole daemon, exactly the failure class WATCHDOG-2/WIFI-WATCHDOG were added to fight. A hung DR script has no deadline and never returns the thread.
- **Recommendation:** Wrap unbounded/potentially-slow shell-outs (dr_scheduler, snapshot, migrate, provision) in tokio::task::spawn_blocking and bound them with proc::output_with_timeout / status_with_timeout_async. Audit the 184 bare call-sites for hot-path ones on the runtime thread.
- **Expected benefit:** Removes runtime-starvation and watchdog-false-restart risk; a hung child is bounded to a timeout instead of forever.
- **Verifier note:** Verified: run_backup (dr_scheduler.rs:111-114) shells `bash -lc` via bare Command::output() with no timeout, called inline at line 208 in async run() on the tokio JoinSet (workers/mod.rs:1083/1110) whose runtime is floored at 4 threads (mackesd.rs:6429-6434), and the watchdog ping — though on a dedicated OS thread — is gated on a runtime-stamped 250ms beat (mackesd.rs:10238-10256) so concurrent blockers can still starve it into a systemd restart; corrections: a single hang pins 1/4 threads and silently wedges DR forever rather than killing the daemon, and the "184 bare call-sites in workers" count is misattributed (that is the whole-src line-match total incl. imports/tests/mitigated wrappers; workers/ has ~65 Command::new sites, ~31 direct .output()/.status(), with 23 fire_and_reap and 24 EFF-20-helper files already mitigated) — pattern real but 3-6x smaller than claimed; category corrected performance→security-reliability since the impact is daemon liveness and silent DR loss.

#### `vdi-vm-4` No client-side reconnect: any transport drop freezes the desktop until the operator manually re-picks
**[P1 · M · risk med]** — `crates/desktop/mde-shell-egui/src/vdi.rs:907-914,941-947,975-981,545-741`

- **Problem:** When any live transport reports Error or Ended, the shell only sets `live_status` and publishes a broker disconnect (vdi.rs:907-914 RDP, 941-947 VNC, 975-981 SPICE); the worker thread (`run_live_rdp`/`run_live_vnc`/`run_live_spice`) has already returned on the first error (e.g. vdi.rs:600-607, 669-676, 735-739), so no further frames arrive and the desktop is frozen on its last frame with a status line. There is no reconnect/backoff anywhere in the shell (grep of vdi.rs for reconnect/retry/respawn/backoff returns nothing). Even RDP's `needs_reconnect` (tier change) and server-initiated Deactivation-Reactivation — which the connect layer explicitly documents as 'callers reconnect' — are not wired to respawn the transport. Design lock 5 promises 'a disconnected VM keeps running; a reconnect returns to Active', and session_roaming::on_node_loss/on_disconnect holds reconnectable mesh state, but nothing on the client actually re-establishes the wire.
- **Why it matters:** Real mesh links drop (roaming Wi-Fi, relay hops, transient VM stalls). A production VDI client must transparently reconnect; here a single blip ends the session and strips the operator's live desktop with no recovery short of navigating back to the Chooser and re-selecting — directly contradicting the advertised roaming/keep-running behavior.
- **Recommendation:** Add a supervised reconnect loop around the transport worker with bounded exponential backoff and a visible 'reconnecting…' state; on RDP `needs_reconnect`/Reactivation, rebuild the connection through the same connect entry point (the tier-change path already exercises this). Re-drive the broker session back to Active on success rather than leaving it Disconnected.
- **Expected benefit:** Sessions survive transient link loss and tier changes, matching the roaming design lock and commercial VDI expectations.
- **Verifier note:** Verified: poll_live_{rdp,vnc,spice} (vdi.rs:907-982) only set live_status + publish broker disconnect while the worker threads permanently return on first error (600-606/669-676/735-738), zero reconnect/retry/backoff hits in vdi.rs, needs_reconnect/Reactivation ("callers reconnect", mde-vdi-rdp connect.rs:27-31) has no shell caller, and design lock 5 (quasar-vdi-desktop.md:156) promises reconnects — one correction: the dock Desktop button (dock.rs:1118 → main.rs:1286 → connect_last_recent) gives a one-click MANUAL reconnect, so recovery is easier than a full Chooser re-pick, but automatic reconnection is entirely absent as claimed.

#### `security-1` Default browser engine (CEF/Chromium) runs untrusted web content with Chromium's own sandbox disabled, no OS sandbox, and no SELinux domain
**[P1 · XL · risk med]** — `crates/desktop/mde-web-cef/src/cef_init.rs:262; crates/desktop/mde-shell-egui/src/web.rs:198-208; crates/mesh/mackesd/Cargo.toml:437; packaging/selinux/ (no CEF .te)`

- **Problem:** The dual-engine browser confines only Servo. THREAT_MODEL.md §3 builds an elaborate confinement (userns + seccomp-bpf + dropped caps + no_new_privs + pivot_root RO rootfs with NO $HOME/keys + cgroups + a confined SELinux domain) — but every layer applies to mde-web-preview (Servo). The CEF engine (mde-web-cef) has NONE of it: `command_line_switches()` unconditionally emits `--no-sandbox` (cef_init.rs:262), disabling even Chromium's own multi-process renderer sandbox; there is no seccomp/userns/pivot_root module anywhere in mde-web-cef (grep for seccomp/unshare/no_new_privs returns nothing); and packaging/selinux ships only mde-web-preview.te/.fc — there is no mde-web-cef SELinux domain. Worse, `preferred_default_engine()` (web.rs:199) returns CEF whenever /usr/bin/mde-web-cef exists and its runtime is present, so CEF is the DEFAULT engine on any node where it is installed, and it ships in the base (Workstation) RPM (Cargo.toml:437). The engine is also a 114-unsafe-block hand-rolled pinned-offset ABI (cef_browser.rs) — a memory-unsafe surface driven by attacker content. THREAT_MODEL §7.4 point 4 and §8.4 both admit CEF 'has no sandbox module' and is 'unaudited here.'
- **Why it matters:** The threat model's own stated assumption is 'assume a determined page can achieve arbitrary code execution inside the helper process' — that is exactly why Servo gets the sandbox. For the DEFAULT engine that assumption is combined with zero confinement: a single malicious/compromised page in a CEF tab executes with the full ambient privileges of the shell/helper process on a node that holds the Nebula CA/identity, ~/.ssh, /etc/nebula, the mesh SSH key, and the age identity. A browser RCE therefore escalates directly to mesh-identity theft with nothing to cross. This directly contradicts the product's central security contract ('the trust boundary is the process boundary').
- **Recommendation:** Interim (cheap): make Servo the default engine and gate CEF behind an explicit, per-node operator opt-in with an in-UI unconfined-engine warning; do not let preferred_default_engine() silently pick an unsandboxed engine. Proper: give CEF the same confinement class as Servo — a real OS sandbox for the mde-web-cef helper (userns/seccomp/pivot_root, drop $HOME/keys from the mount ns) plus a confined SELinux domain (mde-web-cef.te), and stop passing a blanket --no-sandbox (run Chromium's own sandbox where the OS sandbox permits). Add CEF to THREAT_MODEL §1's trust table.
- **Expected benefit:** Removes the largest single escalation path in the product: a browser compromise can no longer read mesh keys/identity. Brings the default engine up to the confinement the security contract already promises for Servo.
- **Verifier note:** Verified every evidence point: cef_init.rs unconditionally passes --no-sandbox in command_line_switches(); web.rs:198-204 preferred_default_engine() picks CEF whenever /usr/bin/mde-web-cef + runtime exist (and the base Workstation RPM ships both the helper at mackesd Cargo.toml:437-438 AND the first-boot mde-cef-runtime-setup.service that auto-provisions the runtime, defeating the 'honest gate'); zero seccomp/unshare/no_new_privs/pivot_root/userns hits in mde-web-cef vs Servo's mde-web-preview/src/sandbox.rs; spawn path (mde-web-preview-client session.rs:670) is a bare Command::new with no external wrapper; packaging/selinux has only mde-web-preview.te/.fc; cef_browser.rs has 114 unsafe occurrences; docs/THREAT_MODEL.md §7.4.4 and §8 themselves admit CEF confinement is absent/unaudited — only correction is the doc lives at docs/THREAT_MODEL.md, and the pinned CEF build (chromium-149.0.7827.201) aging on an airgapped fleet makes n-day renderer exploits directly land as full-process compromise, strengthening the finding.

**[P1 · M · risk low]** — `crates/mesh/mackesd/src/workers/browser_passkeys.rs:1409,1435; crates/desktop/mde-shell-egui/src/web.rs:3491-3510 (handle_passkey_event); crates/mesh/mackesd/src/bin/mackesd.rs:7936`

- **Why it matters:** Same class of bug the EFF-4 catch_unwind in the supervisor was added to fix, but the reconcile thread was never brought under it. A dead-but-reported-alive control worker is a monitoring blind spot.
- **Recommendation:** Either bring reconcile under the Supervisor (via a spawn_blocking-backed adapter so its sync rusqlite stays off the runtime) so it inherits restart/back-off/breaker + honest status, or wrap run_loop in catch_unwind with restart and reflect its liveness truthfully in healthz.
- **Expected benefit:** Reconcile survives a panic and its reported status matches reality.

#### `security-5` Single-use enrollment bearer redemption is a check-then-act TOCTOU race ⚠︎UNVERIFIED
**[P2 · M · risk low]** — `crates/mesh/mackesd/src/nebula_enroll.rs:733 (is_pending check) vs :671 and crates/mesh/mackesd/src/nebula_enroll_endpoint.rs:362 (redeem after delivery); crates/mesh/mackesd/src/bearer_ledger.rs:74,109`

- **Problem:** The 'single-use' join bearer is enforced by two non-atomic filesystem operations: sign_csr_into_bundle checks `bearer_ledger::is_pending()` (a file-exists test, nebula_enroll.rs:733) before signing, and the bearer is only `redeem()`'d (a file remove, bearer_ledger.rs:109) AFTER the bundle is built and delivered — in the network path this happens after the HTTPS response is encoded (nebula_enroll_endpoint.rs:362). Two concurrent POST /enroll requests carrying the same bearer both pass is_pending() before either calls redeem(), so both get signed bundles. redeem() is not guarded by any lock, DB transaction, or atomic claim.
- **Why it matters:** The single-use guarantee (ENT-1) is the entire replay defense on the network enroll path. An attacker who obtains a bearer (or a legitimate holder racing a MITM) can drive concurrent enrollments to mint more than one identity from one token, and can push past the 8-peer cap (the cap count is read before the concurrent signers commit). It weakens the join-token trust model the whole mesh bootstrap rests on.
- **Recommendation:** Make redemption atomic and claim-before-sign: atomically claim the bearer (e.g. rename/unlink the ledger file, or a DB row DELETE...RETURNING under a transaction) BEFORE performing the signing work, and only proceed if the claim succeeded; on any post-claim failure, re-issue rather than leave it spent. Serialize enroll handling per-bearer.
- **Expected benefit:** Restores true single-use semantics and the peer-cap under concurrency; closes a token-replay/identity-duplication window on the public enroll endpoint.

#### `vdi-vm-3` SPICE transport re-copies and re-uploads the entire framebuffer at a fixed 20 Hz with no damage tracking
**[P1 · L · risk med]** — `crates/desktop/mde-vdi-spice/src/connect.rs:227-232; crates/desktop/mde-vdi-spice/src/pixel.rs:173-215; crates/desktop/mde-vdi-spice/src/session.rs:106-125; crates/desktop/mde-shell-egui/src/vdi.rs:728-733`

- **Problem:** `BlockingSpiceTransport::pump_frame` unconditionally sleeps 50ms every iteration (connect.rs:229 `sleep(Duration::from_millis(50))`) — a hard 20 FPS ceiling plus >=50ms added input-to-frame latency. Each pump that returns a surface does three full-frame CPU passes: `Framebuffer::apply_surface` always copies the whole buffer pixel-by-pixel and unconditionally sets `dirty=true` (pixel.rs:196-204 loop; session.rs:112-113), `frame()` then allocates a brand-new full `ColorImage` (pixel.rs:213-215), and the shell re-uploads the whole texture via `TextureHandle::set` (vdi.rs:1075-1082). There is no per-surface change detection: a completely static guest desktop still triggers a full-framebuffer copy + allocation + GPU re-upload 20x/sec. By contrast the VNC path only emits a frame when `rects > 0` (vdi.rs:661-666), confirming SPICE is the outlier. For a 1080p desktop this is ~8MB copied + ~8MB allocated + ~8MB uploaded every 50ms even when nothing changed.
- **Why it matters:** Confirms and quantifies the scoping doc's finding (3 full-frame CPU copies per update, 20Hz poll ceiling). It caps interactivity at 20 FPS, wastes CPU/memory-bandwidth and GPU upload on idle desktops, and drains battery on the physical seats. It is the single biggest efficiency limiter on the SPICE (i.e. the KVM-VM) path.
- **Recommendation:** Drop the fixed 50ms sleep in favor of an event/readiness-driven pump (or a much shorter poll only when the display channel signals a change); hash/dirty-check the decoded surface (or use spice-client damage/rect info) so `apply_surface` and the texture upload are skipped when unchanged; reuse the ColorImage allocation instead of allocating per frame; and upload only dirty sub-rectangles where egui/spice-client allow. At minimum, gate the frame emission on an actual content change as VNC already does.
- **Expected benefit:** Higher and adaptive frame rate, lower input latency, and near-zero cost on idle desktops — materially better VDI responsiveness and battery life.
- **Verifier note:** Verified every cited line: the unconditional 50ms sleep in BlockingSpiceTransport::pump_frame (connect.rs:229), the full-frame pixel copy with no change detection (pixel.rs:196-204), the unconditional dirty=true (session.rs:112), the per-frame ColorImage allocation (pixel.rs:213-215), the ungated Frame emission on Ok(true) vs VNC's rects>0 gate (vdi.rs:728-733 vs 661-666), and the full TextureHandle::set upload (vdi.rs:1075-1082); sole caveat is that spice-client 0.2.0's get_display_surface source isn't vendored locally, but the in-repo doc ("false if the channel has no surface yet") plus zero in-repo change-gating support the static-desktop 20Hz full-recopy claim, and the 20 FPS cap/latency/triple-full-frame-pass findings hold regardless.

#### `perf-1` DRM present loop renders the full egui UI unconditionally at panel refresh rate — all repaint throttling in the shell is dead code on the production seat
**[P1 · L · risk med]** — `crates/shared/mde-egui/src/drm.rs:854-1157 (run_drm present loop)`

- **Problem:** The production DRM runner's `while !quit` loop (drm.rs:854) unconditionally executes libinput dispatch -> `egui_ctx.run()` (line 1092) -> tessellate -> paint -> `swap_buffers` -> page-flip on EVERY iteration, paced only by the vblank wait (lines 1140-1150). It never sleeps waiting for input, never inspects egui's repaint scheduling (`grep repaint drm.rs` returns zero hits), and never skips a frame when nothing changed. Every `ctx.request_repaint_after(REFRESH)` call across the shell (chat.rs:390, datacenter.rs:658, system.rs:261, etc.) is therefore a no-op on the DRM seat — the canonical deployment. Additionally each frame calls `gbm.add_framebuffer` + `destroy_framebuffer` (drm.rs:1124-1153) instead of caching FB handles per GBM buffer object.
- **Why it matters:** A fully idle desktop performs a complete immediate-mode layout/tessellation/GL paint of the entire shell (dock, taskbar, surfaces) 60 times per second, forever. On the thin-client/tablet hardware this OS targets (Surface-class devices with battery — drm.rs:769 explicitly supports them) this is a continuous ~1-core CPU + GPU load, with direct battery, thermal, and fan consequences. It also amplifies every other per-frame inefficiency (perf-8, perf-9) by running them at 60Hz instead of on-demand.
- **Recommendation:** Make the loop event-driven: poll(2) on the libinput fd + a timerfd armed from egui's `FullOutput` repaint delay (eframe's exact contract), rendering only on input, expired repaint deadline, or an explicit wake. Separately, cache the DRM framebuffer handle per GBM BO (BO userdata is the standard idiom) instead of add/destroy per frame. The existing per-surface `request_repaint_after` calls then start actually working.
- **Expected benefit:** Idle CPU drops from a continuously loaded core to near-zero; battery life and thermals on tablet/laptop seats improve dramatically; no visible behavior change since animations already call request_repaint.
- **Verifier note:** Verified in drm.rs: the run_drm loop (854-1157) has no wait primitive (grep sleep/poll/timerfd = zero hits), unconditionally runs egui_ctx.run at 1092 paced only by the page-flip vblank drain (1140-1150), ignores FullOutput's repaint_delay (grep repaint = zero hits vs 47 request_repaint sites in mde-shell-egui incl. chat.rs:390/system.rs:261/datacenter.rs:658), and does add_framebuffer/destroy_framebuffer per frame (1124/1153) — and main.rs:1565 confirms run_drm is the production seat path, so idle seats permanently render the full shell at refresh rate on battery Surface-class targets; P1 stands (severe pervasive power/perf waste, not a correctness/availability failure).

#### `browser-7` CEF pending resource-verdict map is unbounded with no verdict timeout; idle-suspended tabs stall in-flight loads and pin CEF callbacks

#### `perf-8` Browser panel clones the full frame image on every paint-ready (~8MB per 1080p frame) just to retain a CPU-side copy
**[P2 · S · risk low]** — `crates/desktop/mde-shell-egui/src/web.rs:4061-4066`

- **Verifier note:** Verified: web.rs:4062 deep-clones the owned ColorImage from take_frame (session.rs:363) on every frame before texture upload, all last_frame consumers (web.rs:2617-2792, 3372) are on-demand capture verbs, and the Arc fix is directly supported since epaint 0.31.1 has ImageData::Color(Arc<ColorImage>) + From<Arc<ColorImage>> (image.rs:15,249) — a real ~8MB/frame render-thread memcpy tax with a trivial low-risk fix.
