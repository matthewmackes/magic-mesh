# Cloud backend — OpenTofu provision (`infra/tofu/cloud/`)

**WL-ARCH-001 Phase B.** The OpenTofu half of the cloud backend that **replaces
the deleted OpenStack control plane**. OpenTofu *provisions* local libvirt/KVM
VMs + their network declaratively (the Heat/Nova/Neutron replacement); **Ansible**
(`automation/ansible/`) then *configures* each VM over the mesh-derived dynamic
inventory. Local-first (E12) — no external cloud.

The mackesd `cloud` worker (`crates/mesh/mackesd/src/workers/cloud.rs`) shells
this root (`tofu -chdir=infra/tofu/cloud plan/apply`) behind the
`MDE_CLOUD_APPLY=1` operator gate and publishes `state/cloud/<node>`.

## Layout
```
versions.tf     provider pins — dmacvicar/libvirt ~> 0.8.3 (+ external/null)
providers.tf    the libvirt connection (var.libvirt_uri)
backend.tf      etcd http state backend (/tofu/state/cloud), like edgeos/xen-xapi
variables.tf    libvirt_uri · pool · base image · network · vms · mesh-join · secrets
main.tf         the network module + a base volume + a vm module per var.vms key
secrets.tf      the mde-seal → IaC bridge (external data source; item 3)
outputs.tf      network_id + the instances roster (neutral CloudInstance shape)
modules/network the libvirt_network on the Nebula-adjacent bridge (replaces Neutron)
modules/vm      a CoW root volume + mesh-join cloud-init disk + the libvirt_domain
cloud-init/     the mesh-join cloud-init template (SEC-001 join path)
scripts/        mde-seal-external.sh — the tofu external-data secrets bridge
```

## State backend (SUBSTRATE-V2)
Joins the etcd state plane at `/tofu/state/cloud`. The tracked `backend.tf`
carries only the static lock/unlock methods — the per-mesh address is supplied at
init via `-backend-config=cloud.backend.hcl` (generated by
`automation/state-backend/gen-backend-config.sh`, gitignored), so no LAN-node
literal lives in a tracked `.tf`.

## Secrets (mde-seal, NO Vault)
The one secret this root consumes — the Nebula join/enrollment token baked into
each VM's cloud-init — is age-sealed in the mesh store and resolved at apply time
by `scripts/mde-seal-external.sh` (an OpenTofu `external` data source shelling
`mcnf-secret.sh get <name>`). The SAME store bridges to Ansible via
`automation/ansible/plugins/lookup/mde_seal.py` — one secret system, no Ansible
Vault (WL-ARCH-001 decided-stack #8). The unsealed value is marked `sensitive`
everywhere it flows. Self-test the bridge offline:
```sh
bash scripts/mde-seal-external.sh --selftest   # stubbed store, no live secrets
```

## Usage
```sh
cd infra/tofu/cloud
cp terraform.tfvars.example terraform.tfvars   # edit the VM set
source ./env.sh                                # (see env.sh.example)

# CI / offline — validate + fmt only, no state, no live libvirt:
tofu init -backend=false && tofu validate && tofu fmt -check -recursive

# Live (operator, Phase D) — against a real hypervisor + the etcd backend:
tofu init -backend-config=cloud.backend.hcl
tofu plan          # or `tofu apply` past the MDE_CLOUD_APPLY gate
tofu output instances
```

## Provider pin
`dmacvicar/libvirt ~> 0.8.3` — the stable classic-block schema (`disk {}` /
`network_interface {}` / `graphics {}`). The 0.9.x line rewrote domain devices
into a single `devices` attribute (a different, still-settling contract), so this
root deliberately stays on 0.8.x; `.terraform.lock.hcl` is committed.
